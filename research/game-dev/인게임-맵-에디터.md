# 인게임 맵 에디터

조사 일자: 2026-03-12

조사 일자: 2026-03-12
목적: 게임 내에 내장되는 맵 에디터 구현 방안
출처: 일반적 게임 개발 패턴, Tiled/LDtk 구조 분석

---

## 1. 핵심 요약

**인게임 맵 에디터란:** 게임 실행 중에 플레이어나 개발자가 맵을 수정할 수 있는 기능

**주요 접근 방식:**
1. **완전한 에디터 내장** — Tiled/LDtk 스타일의 에디터를 게임 내에 구현
2. **간소화된 에디터** — 플레이어용으로 제한된 기능 제공
3. **외부 툴 통합** — Tiled/LDtk JSON을 로드하여 게임 내에서 편집

**실용적 추천:**
- **개발자용:** 외부 툴(Tiled/LDtk) + JSON 임포트
- **플레이어용:** 간소화된 인게임 에디터
- **하이브리드:** 개발은 외부 툴, 플레이어는 인게임 에디터

## 2. 상세 분석

### 2.1 아키텍처 패턴

#### Pattern A: Tile-Based Grid System

**구조:**
```
Game Engine
├─ MapData (2D array / sparse array)
│   ├─ width, height
│   ├─ tiles[layer][y][x]
│   └─ entities[]
├─ TileRenderer
│   ├─ tileset sprite atlas
│   └─ render tiles to canvas/webgl
├─ EditorMode
│   ├─ brush system
│   ├─ layer management
│   └─ undo/redo stack
└─ SaveSystem
    ├─ JSON export
    └─ localStorage / server
```

**데이터 구조:**
```typescript
interface TileMap {
  width: number;
  height: number;
  tileSize: number;
  layers: Layer[];
  entities: Entity[];
}

interface Layer {
  id: string;
  name: string;
  type: 'tile' | 'object';
  visible: boolean;
  opacity: number;
  data: number[][]; // tile IDs, -1 for empty
}

interface Entity {
  id: string;
  type: string;
  x: number;
  y: number;
  properties: Record<string, any>;
}
```

**장점:**
- 단순하고 이해하기 쉬움
- 대부분의 2D top-down 게임에 적합
- 저장/로드가 직관적

**단점:**
- 큰 맵에서 메모리 사용량 증가
- 동적 변경에 제약

---

#### Pattern B: Chunked Infinite Map

**구조:**
```
MapManager
├─ chunks: Map<string, Chunk>
├─ activeChunks: Set<Chunk>
└─ chunkLoader/Generator

Chunk {
  x, y (chunk coordinates)
  tiles: Uint8Array
  entities: Entity[]
  isDirty: boolean
}
```

**장점:**
- 무한 맵 지원
- 메모리 효율적 (필요한 청크만 로드)
- 점진적 저장 가능

**단점:**
- 구현 복잡도 증가
- 경계 처리 필요

---

#### Pattern C: Entity-Component System

**구조:**
```
ECS World
├─ TileComponent (position, tileId)
├─ PositionComponent (x, y)
├─ RenderableComponent (sprite, layer)
└─ EditorComponent (isLocked, properties)
```

**장점:**
- 유연한 확장성
- 게임 로직과 에디터 로직 통합
- 복잡한 객체 관리 용이

**단점:**
- 오버헤드
- 단순한 타일 맵에는 과함

### 2.4 Undo/Redo 시스템

#### Command Pattern

```typescript
interface Command {
  execute(): void;
  undo(): void;
  redo(): void;
}

class PaintTilesCommand implements Command {
  private tiles: {x: number, y: number, oldTile: number, newTile: number}[];

  execute() {
    this.tiles.forEach(t => {
      map.setTile(t.x, t.y, t.newTile);
    });
  }

  undo() {
    this.tiles.forEach(t => {
      map.setTile(t.x, t.y, t.oldTile);
    });
  }

  redo() {
    this.execute();
  }
}

class EditorHistory {
  private undoStack: Command[] = [];
  private redoStack: Command[] = [];
  private maxSize = 100;

  execute(command: Command) {
    command.execute();
    this.undoStack.push(command);
    this.redoStack = []; // clear redo on new action
    if (this.undoStack.length > this.maxSize) {
      this.undoStack.shift();
    }
  }

  undo() {
    const command = this.undoStack.pop();
    if (command) {
      command.undo();
      this.redoStack.push(command);
    }
  }

  redo() {
    const command = this.redoStack.pop();
    if (command) {
      command.redo();
      this.undoStack.push(command);
    }
  }
}
```

**주의사항:**
- 메모리 관리 (최대 개수 제한)
- 대량 변경은 단일 커맨드로 묶기
- 세이브 포인트 마킹

### 2.5 실시간 협업 (옵션)

#### Operational Transformation

```
Client A: Paint (10, 10) → Tile 5
Client B: Paint (10, 10) → Tile 7

Server transforms:
- A's operation: (10, 10) → 5
- B's operation: (10, 10) → 7 (conflict!)

Resolution: Last-Write-Wins or Merge
```

**구현 포인트:**
- 각 변경사항에 타임스탬프/버전 부여
- 변경사항을 작은 단위로 분할
- 충돌 감지 및 해결 정책

### 4.4 픽셀 오피스에 적용

**추천 접근:**

1. **개발 단계:** Tiled로 기본 오피스 맵 제작
2. **런타임:** Tiled JSON 로드
3. **플레이어 편집:** 간소화된 인게임 에디터 (가구 배치, 벽 수정)

**이유:**
- agent 활동 시각화가 주 목적
- 맵 자체는 정적일 가능성 높음
- 플레이어가 agent 워크스페이스 커스터마이징 정도면 충분

**구현 범위:**
- 에디터 모드 토글
- 가구/오브젝트 배치
- 벽/바닥 타일 변경
- 저장 (localStorage)
- Tiled JSON 임포트 (개발자용)
