# Karpathy의 LLM Wiki - 개인 지식베이스 패턴

조사 일자: 2026-04-08

## 핵심 요약

Andrej Karpathy가 제안한 LLM Wiki는 LLM이 개인 지식베이스를 자동으로 구축하고 유지관리하는 패턴이다. RAG와 달리 매번 검색하는 방식이 아니라, LLM이 지속적으로 위키를 컴파일하고 정제한다.

---

## 1. 개념

### 1.1 RAG와의 차이점

| 구분 | RAG | LLM Wiki |
|------|-----|----------|
| 작동 방식 | 질문 시마다 관련 청크 검색 | 미리 구조화된 위키에서 조회 |
| 지식 축적 | 없음 (매번 새로 검색) | 있음 (지속적으로 컴파일) |
| 교차 참조 | 실시간으로 찾아야 함 | 이미 연결되어 있음 |
| 모순 탐지 | 어려움 | 이미 플래그되어 있음 |
| 유지관리 | 없음 | LLM이 자동 수행 |

### 1.2 핵심 아이디어

> "RAG는 매번 지식을 새로 발견한다. LLM Wiki는 지식을 한 번 컴파일하고 계속 유지한다."

LLM이 원시 문서를 읽고, 요약하고, 교차 참조하고, 위키에 통합한다. 인간은 소싱과 질문에만 집중하면 된다.

## 2. 아키텍처

### 2.1 세 가지 레이어

```
┌─────────────────────────────────────────┐
│  Raw Sources (원시 자료)           [[]()]()      │
│  - 불변 (읽기 전용)                      │
│  - 논문, 기사, 이미지, 데이터 파일        │
└─────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────┐
│  Wiki (위키)                             │
│  - LLM이 생성하고 유지관리               │
│  - 마크다운 파일들                       │
│  - 요약, 엔티티 페이지, 개념 페이지       │
└─────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────┐
│  Schema (스키마)                         │
│  - CLAUDE.md 또는 AGENTS.md              │
│  - LLM에 위키 구조와 규칙을 알려줌        │
│  - 워크플로우 정의                       │
└─────────────────────────────────────────┘
```

### 2.2 핵심 파일

| 파일 | 역할 |
|------|------|
| index.md | 콘텐츠 중심 카탈로그. 모든 페이지 링크와 요약 |
| log.md | 시간 순서 기록. 수집, 쿼리, 린트 기록 |

### 2.3 디렉토리 구조 예시

```
wiki/
├── _templates/
├── projects/
├── research/
├── reference/
├── meetings/
├── inbox/
├── index.md
└── log.md
```

## 3. 워크플로우

### 3.1 Ingest (수집)

```
원시 문서 → LLM이 읽음 → 대화로 핵심 확인
    → 요약 페이지 작성 → index.md 업데이트
    → 관련 엔티티/개념 페이지 업데이트 → log.md에 기록
```

- 하나의 소스가 10-15개 위키 페이지에 영향
- 인간이 가이드하며 진행 (또는 배치 처리)

### 3.2 Query (질의)

```
질문 → LLM이 관련 페이지 검색 → 읽고 종합 → 답변 + 인용
    → 좋은 답변은 위키에 새 페이지로 저장
```

**중요한 통찰:**
> 좋은 답변은 채팅 기록에 사라지지 않고 위키에 다시 저장되어야 한다. 이렇게 하면 탐색이 지식베이스에 누적된다.

### 3.3 Lint (정기 점검)

LLM에게 정기적으로 위키 건강 체크 요청:

- 페이지 간 모순 탐지
- 오래된 주장 (새 소스에 의해 대체됨)
- 고아 페이지 (인바운드 링크 없음)
- 중요 개념인데 페이지 없음
- 누락된 교차 참조
- 웹 검색으로 채울 수 있는 데이터 갭

## 4. 사용 사례

### 4.1 개인

- 목표, 건강, 심리학, 자기계발 추적
- 일지, 기사, 팟캐스트 노트를 구조화된 자화상으로 축적

### 4.2 리서치

- 주제 깊이 파고들기 (수주~수개월)
- 논문, 기사, 보고서를 읽으며 종합 위키 구축

### 4.3 독서

- 챕터별로 기록
- 캐릭터, 테마, 플롯 스레드 페이지 구축
- 팬 위키(예: Tolkien Gateway)를 개인적으로 구축

### 4.4 비즈니스/팀

- Slack 스레드, 회의록, 프로젝트 문서, 고객 통화로 위키 유지
- 팀원은 리뷰만 하면 됨

### 4.5 기타

- 경쟁 분석, 실사(Due Diligence), 여행 계획, 강의 노트, 취미 딥다이브

## 6. 마크다운이 적합한 이유

### 6.1 이식성과 미래 보장

- .md 파일은 텍스트 파일
- 모든 OS, 모든 에디터에서 영원히 열림
- 특정 회사/앱에 종속되지 않음

### 6.2 LLM이 네이티브로 읽음

- Claude는 GitHub README, 문서 사이트 등 방대한 마크다운으로 훈련됨
- 헤더, 리스트, 코드 블록, 볼드를 구조로 인식

### 6.3 명확성 강제

- 헤더는 섹션 이름 필요
- 리스트는 항목 분리 필요
- 형식이 명확성을 유도

### 6.4 잠금 없음

- git으로 동기화
- VS Code에서 열기
- Obsidian에서 보기
- GitHub에 push
- 터미널에서 읽기

## 7. 모범 사례

### 7.1 요약 작성

- 각 노트 상단에 한 줄 요약 필수
- Claude가 전체 파일을 읽지 않고도 관련성 판단 가능

### 7.2 일관된 용어 사용

- "RAG"와 "retrieval augmented generation" 혼용보다는 하나로 통일
- 필요하면 별칭 라인 추가

### 7.3 노트 간 연결

- Obsidian `[[wiki links]]` 형식 사용
- 연결된 위키는 평면 파일보다 더 풍부한 그래프 제공

### 7.4 노트 분할

- 10,000단어 통합 문서보다 10개의 1,000단어 노트가 조회에 유리
- 파일이 구체적일수록 정확한 검색 가능

### 7.5 /inbox 패턴

- 완벽하지 않아도 됨
- /inbox에 임시 저장 후 Claude에게 정리 요청

## 8. 철학

### 8.1 인간의 역할 vs LLM의 역할

| 인간 | LLM |
|------|-----|
| 소스 큐레이션 | 요약 |
| 분석 방향 지시 | 교차 참조 |
| 좋은 질문하기 | 파일링 |
| 의미 해석 | 북키퍼핑 |

### 8.2 유지관리 부담 해결

> 위키를 포기하는 이유는 읽거나 생각해서가 아니라, 북키퍼핑 때문이다. 교차 참조 업데이트, 요약 최신화, 모순 메모, 수십 페이지의 일관성 유지. 인간은 유지관리 부담이 가치보다 빨리 커져서 위키를 포기한다. LLM은 지루하지 않고, 교차 참조를 잊지 않고, 한 번에 15개 파일을 수정할 수 있다.

### 8.3 Memex와의 연결

Vannevar Bush의 Memex (1945)와 정신적으로 연결됨:

- 개인, 큐레이션된 지식 저장소
- 문서 간 연결이 문서 자체만큼 가치 있음
- Bush가 해결하지 못한 것: **누가 유지관리하는가?**
- LLM이 그 역할을 맡음
