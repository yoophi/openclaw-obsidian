# Hermes Agent (스스로 진화하는 에이전트) 정리

- 출처: https://javaexpert.tistory.com/1673
- 원문 제목: Hermes Agent (스스로 진화하는 에이전트)
- 작성일: 2026-03-24

## 한줄 요약

이 글은 Hermes Agent를 단순한 "기억하는 AI"가 아니라, 스킬 축적, 세션 검색, 다중 실행 환경, 메시징 게이트웨이, 크론 자동화, 연구용 trajectory 수집까지 포함한 장기 실행형 에이전트 플랫폼으로 해석합니다.

## 핵심 주장

글의 핵심 논지는 다음과 같습니다.

1. Hermes는 단순 챗봇이나 LLM 래퍼가 아니다.
2. memory보다 skill 시스템이 더 본질적인 차별점이다.
3. session search, prompt stability, environment persistence 같은 운영 요소가 실제 장기 협업에 중요하다.
4. CLI와 메신저, 스케줄러를 하나의 공통 코어 위에 올려 "항상 켜진 에이전트"에 가깝게 설계되어 있다.
5. 사용자용 도구이면서 동시에 trajectory 수집과 RL 연계를 고려한 research-ready 플랫폼이다.

## 글에서 정리한 Hermes Agent의 특징

### 1. "기억하는 챗봇"이 아니라 운영형 에이전트 플랫폼

글은 Hermes를 "기억을 가진 에이전트" 정도로 보면 절반만 본 것이라고 설명합니다. 실제로는 다음 계층이 함께 묶인 운영형 시스템이라고 봅니다.

- 에이전트 런타임
- 툴 실행 계층
- 기억/스킬 계층
- 메시징 게이트웨이
- 스케줄러
- 연구용 데이터 생성 계층

즉 핵심은 대화 자체보다, 여러 인터페이스에서 오래 실행되며 작업 경험을 시스템적으로 축적하는 구조에 있다는 해석입니다.

### 2. 장기 사용성 문제를 정면으로 다룸

글은 기존 에이전트 프레임워크의 한계로 "세션을 넘는 연속성 부족"을 지적합니다.

Hermes는 이를 보완하기 위해 다음 요소를 강조한다고 정리합니다.

- persistent memory
- session search
- user profile / Honcho 계층
- skills

즉 어제 했던 작업, 실패한 시도, 사용자 선호, 특정 프로젝트의 관습을 이어받는 능력이 Hermes의 중요한 설계 목표라고 봅니다.

### 3. memory보다 skill이 더 중요함

이 글에서 가장 강조하는 포인트 중 하나는 memory와 skills의 구분입니다.

- memory: 사용자 환경, 프로젝트, 선호, 설치된 도구, 기술 스택 같은 지속적 사실 정보
- skills: 작업 과정에서 발견한 재사용 가능한 절차적 지식

예시로 글은 skill을 이런 것으로 설명합니다.

- 특정 저장소에서 릴리스 노트 만드는 절차
- PDF에서 필요한 내용을 추출하는 워크플로
- 특정 배포 환경에서 로그를 모으고 문제를 진단하는 단계
- 외부 CLI/API를 다루는 실행 절차

즉 Hermes의 진짜 강점은 "정보 저장"보다 "작업 경험을 절차로 굳히는 것"에 있다는 해석입니다.

### 4. memory 시스템은 의도적으로 보수적임

글은 Hermes memory 시스템이 단순히 메모리를 계속 프롬프트에 덧붙이는 방식이 아니라는 점을 중요한 설계 철학으로 봅니다.

요지는 다음과 같습니다.

- 세션 시작 시 memory가 frozen block처럼 주입됨
- 세션 중 새 메모리가 저장되어도 현재 세션 프롬프트는 즉시 바뀌지 않음
- 다음 세션에서 반영됨

이걸 글은 prefix cache 보존과 prompt stability를 위한 현실적인 선택으로 해석합니다.

즉 Hermes는 "항상 최신 메모리 즉시 반영"보다,
- 프롬프트 안정성
- 캐시 효율
- 비용/성능 균형
을 더 중요하게 보는 설계라고 설명합니다.

### 5. session search는 벡터 DB보다 운영형 검색에 가깝다

글은 Hermes의 session search를 "거대한 의미 검색 시스템"보다는 SQLite FTS5 기반의 실용적인 기록 검색으로 봅니다.

장점으로 보는 부분은 다음과 같습니다.

- 지난번 정확히 어떤 커맨드를 쳤는지 찾기 쉬움
- 어떤 실패 로그가 있었는지 찾기 쉬움
- 특정 날짜/서버/결정 이력을 다시 찾기 좋음
- 그 결과를 LLM이 요약해 cross-session recall에 활용 가능

즉 Hermes는 의미적 유사도만 추구하기보다, 실제 운영 기록의 정확한 재탐색을 더 중시하는 구조라는 분석입니다.

### 6. 아키텍처 핵심은 단일 채팅 루프가 아님

글은 Hermes를 단순한 "OpenAI compatible chat loop + some tools"로 보면 안 된다고 강조합니다.

구조적으로 다음 계층을 함께 보는 것이 맞다고 정리합니다.

- `run_agent.py`: 핵심 에이전트 루프
- `cli.py`: 인터랙티브 터미널 UI
- `model_tools.py`: tool discovery / orchestration
- `hermes_state.py`: SQLite 기반 세션/상태 DB
- `gateway/`: 메시징 게이트웨이
- `cron/`: 스케줄러
- `skills/`: 스킬 계층
- `environments/`: RL / benchmark 환경

즉 CLI, gateway, cron, ACP 같은 입력 채널이 여러 개여도, 아래에는 공통 코어가 있다는 점을 핵심으로 봅니다.

### 7. 툴보다 실행 환경 설계가 중요함

글은 Hermes의 tool runtime에서 "무슨 툴이 있느냐"보다 "어디서 실행되느냐"가 더 중요하다고 봅니다.

지원 백엔드 예시:

- local
- Docker
- SSH
- Singularity
- Modal
- Daytona

해석 포인트는 다음과 같습니다.

- 로컬은 빠르지만 위험할 수 있음
- Docker는 격리와 재현성에 유리
- SSH는 에이전트를 자기 머신에서 분리할 수 있어 보안상 장점
- Modal/Daytona는 원격 지속 환경 제공

즉 Hermes는 기억만 지속하는 것이 아니라, 실행 환경 자체의 지속성까지 고려하는 플랫폼이라는 분석입니다.

### 8. delegation과 병렬화는 진짜 멀티 에이전트 지향

글은 Hermes가 단순 역할놀이형 멀티 에이전트가 아니라,
- 격리된 subagent
- 병렬 workstream
- Python pipeline / RPC 기반 툴 호출
을 명시적으로 다룬다고 설명합니다.

즉 긴 작업을 한 세션 안에 계속 쌓는 대신,
- 작업 일부를 분리 실행하고
- 결과만 회수하는 방식으로
컨텍스트 비용과 실패 범위를 줄이는 구조라고 해석합니다.

### 9. 보안 모델을 별도 설계 요소로 다룸

글은 Hermes가 "실행 가능한 에이전트"라는 점을 보안 설계에서 분명히 인식하고 있다고 설명합니다.

언급된 보안 축은 다음과 같습니다.

- 누가 에이전트와 대화할 수 있는지
- 위험한 명령 승인 플로
- 컨테이너 격리
- MCP 자격증명 필터링
- 프로젝트 파일 내 prompt injection 스캔

즉 Hermes는 단순히 답변만 하는 모델이 아니라 실제 커맨드를 실행하고 시스템을 바꾸는 존재이기 때문에, 보안 경계를 아키텍처 수준에서 다룬다는 점을 강조합니다.

### 10. gateway + cron = 항상 켜진 에이전트

글은 Hermes의 실전적 가치로 다음 조합을 높게 평가합니다.

- CLI
- messaging gateway
- cron scheduler

이를 통해 Hermes는 다음과 같은 패턴을 가능하게 한다고 봅니다.

- 원격 환경에서 계속 살아 있음
- Telegram/Discord 등으로 호출 가능
- 예약 작업 수행 가능
- 결과를 특정 플랫폼으로 배달 가능

즉 "필요할 때 잠깐 쓰는 도구"가 아니라, 백그라운드에서 계속 동작하는 에이전트 프로세스에 가깝다는 해석입니다.

### 11. 연구 플랫폼으로서의 의미

글은 Hermes가 user-facing agent이면서 동시에 research-ready platform이라고 봅니다.

연구 측면 요소:

- batch trajectory generation
- trajectory compression
- RL environments
- Atropos integration
- tool-calling 모델을 위한 데이터 생성

즉 Hermes는 잘 쓰는 앱인 동시에, 더 나은 에이전트를 훈련하기 위한 실험장 역할도 하도록 설계되었다는 해석입니다.

## 이 글의 최종 해석

글이 내리는 가장 중요한 결론은 다음과 같습니다.

> Hermes는 "기억하는 챗봇"이 아니라, 오래 실행되며 경험을 스킬화하는 에이전트 런타임이다.

그리고 이 프로젝트의 본질은 다음 6가지 축으로 요약됩니다.

- 사실 기억과 절차 지식의 분리
- prompt stability를 고려한 보수적 memory 반영
- FTS5 기반의 운영형 session recall
- 실행 환경 자체를 1급 요소로 다루는 구조
- CLI, gateway, cron, ACP를 하나의 코어 위에 올린 아키텍처
- 실사용과 연구 데이터 생성을 연결하는 플랫폼 지향성

## 내 관점에서 다시 요약

이 글은 Hermes를 단순한 AI 챗봇 진화판으로 보지 말고,
다음과 같은 질문에 대한 하나의 답으로 읽으라고 제안합니다.

- 에이전트를 장기 협업 도구로 만들려면 무엇이 필요한가?
- 단발성 자동화가 아니라 운영 가능한 에이전트 시스템이 되려면 어떤 층이 필요한가?
- 경험을 기억하는 것과 절차로 굳히는 것은 어떻게 다른가?

글의 답은 명확합니다.

- 기억해야 하고
- 검색할 수 있어야 하며
- 스킬로 굳어져야 하고
- 안전하게 실행돼야 하며
- 메신저와 터미널을 넘나들 수 있어야 하고
- 필요하면 스스로 스케줄까지 수행해야 한다

## 실무적으로 참고할 포인트

이 글을 바탕으로 Hermes를 볼 때 특히 유용한 체크포인트는 다음과 같습니다.

1. memory 기능만 보지 말고 skill 축적 구조를 같이 볼 것
2. CLI 전용 도구가 아니라 gateway + cron 조합까지 포함해 평가할 것
3. session recall이 벡터 검색이 아니라 운영형 기록 검색이라는 점에 주목할 것
4. tool 수보다 실행 환경 backend 설계가 더 중요할 수 있음을 볼 것
5. 연구용 trajectory 수집 플랫폼으로서의 가능성도 함께 볼 것
