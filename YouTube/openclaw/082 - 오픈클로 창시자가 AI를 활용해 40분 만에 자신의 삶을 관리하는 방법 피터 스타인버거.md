## How OpenClaw’s creator uses an AI that lives on his computer

Peter Steinberger demonstrates a personal AI assistant (installed from cloud.bot / OpenClaw) that he connects to messaging apps to run real tasks for him. It reads tweets and bug reports, checks out repositories, applies fixes, commits changes and posts updates; it finds and converts audio files with ffmpeg, calls transcription APIs, and replies to voice messages. Because the agent has access to local files, CLIs and browsers, it can control devices, check flights, handle bookings, read email/calendars, create GitHub issues, manage passwords in its own vault and even adjust smart‑home gear—all via chat or a short command sequence. *If you give an AI access to your computer, they can basically do anything that you can do.*

Peter describes building lightweight, hackable installs (terminal one‑liners or full checkouts) so the agent can read and even reconfigure its own harness; users expose APIs and keys to enable integrations. He emphasizes persistent memory and skills: after the agent learns a website’s quirks it can repeat multi‑step tasks (like airline check‑in) far faster, then create reusable skills so future runs take minutes. He also highlights common, safe starter uses—calendar and email help, reminders, sleep/fitness nudges, bookmarks→todo capture, family chat automation—and predicts these assistants will replace many single‑purpose phone apps. *This will blend away probably 80% of the apps that you have on your phone.*

Peter warns about real risks and common developer traps: an agent with full access can delete files or lock systems if you mindlessly confirm prompts, so boundaries and user judgment remain essential. He pushes back on “agent orchestration” vanity projects that run dozens of agents for hours with little product sense, calling that the “agentic trap”; good results still require human taste, iteration and clear vision. His pragmatic workflow: play with the model to learn its language, iterate in short human–AI loops, ask clarifying questions, treat community pull requests as prompt requests, and prefer small, testable features rather than massive unattended automations. The talk mixes demos, warnings and practical tips for anyone curious whether an always‑available, resourceful assistant is worth trying.
